Documentação do Projeto

Visão técnica e de negócios para investidores, desenvolvedores e stakeholders.

Product Requirements Document (PRD)

1. Visão Geral

O Licitus Bot é uma plataforma SaaS B2B que automatiza a participação em pregões eletrônicos. Seu objetivo é eliminar a perda de licitações por lentidão humana, garantindo o melhor preço possível dentro da margem de lucro estipulada pelo usuário.

2. Problema a ser resolvido

  • Empresas perdem pregões na fase de "tempo randômico" ou em disputas acirradas por não conseguirem digitar e enviar lances rápido o suficiente.
  • Erro humano ao calcular a margem durante o calor da disputa, resultando em lances abaixo do custo (prejuízo).

3. Público-Alvo

Pequenas e Médias Empresas (PMEs) que fornecem para o Governo Federal via portal Compras.gov.br e participam de licitações de alta concorrência.

4. Funcionalidades Core (User Stories)

  • Como usuário, quero cadastrar minha margem mínima para que o bot pare de dar lances e não me gere prejuízo.
  • Como usuário, quero definir um "decremento" (ex: cobrir a oferta em R$ 0,05) para otimizar meu lucro.
  • Como usuário, quero um dashboard para acompanhar os itens que estou ganhando e perdendo em tempo real.

Non-Product Requirements (N-PRD) & Arquitetura

1. Requisitos de Desempenho

  • Latência: O tempo de resposta entre a leitura de um novo lance concorrente e o envio da contraproposta deve ser inferior a 500ms.
  • Disponibilidade: Uptime de 99.9% durante o horário comercial (08h às 18h), período crítico de pregões.

2. Requisitos de Segurança

  • Criptografia: As credenciais do usuário (Gov.br) devem ser armazenadas com criptografia AES-256 no banco de dados.
  • Conexão: Toda a comunicação cliente-servidor deve ocorrer via TLS 1.3 (HTTPS).

3. Requisitos de Conformidade Legal

O bot não pode sobrecarregar os servidores do governo. Deve implementar "rate limiting" (ex: requisições a cada X segundos baseados na regra do portal) para não caracterizar ataque de negação de serviço (DDoS) ou fraude em licitação.

4. Stack Tecnológica Sugerida

  • Backend/Bot Worker: Python (Selenium/Playwright para scraping complexo ou HTTPX para chamadas de API nativas).
  • Frontend (Dashboard): React.js ou Vue.js com TailwindCSS.
  • Banco de Dados: PostgreSQL para dados relacionais, Redis para fila de lances em tempo real (Celery/RabbitMQ).
  • Infraestrutura: AWS (EC2/Fargate) com IPs rotativos (Proxies) se necessário.

Escopo do MVP (Minimum Viable Product)

Objetivo do MVP: Validar se a tecnologia consegue autenticar, ler o menor preço e enviar um lance válido mais rápido que um humano, apenas no portal mais popular (Compras.gov.br).

O que ENTRA no MVP

  • Suporte exclusivo ao portal Compras.gov.br.
  • Modo de disputa: Apenas "Aberto".
  • Estratégia de Lance Única: Cobrir o primeiro colocado por um valor fixo (ex: R$ 0,01 a menos).
  • Cadastro simples de item com: Número do Pregão, Item, Valor Limite Mínimo.
  • Botão de "Start/Stop" manual por item.
  • Script rodando localmente (Desktop App ou CLI em Python) para evitar custos iniciais com servidores em nuvem.

O que FICA DE FORA (Backlog V2)

  • Suporte a outros portais (Licitações-e, BLL, BEC).
  • Modo de disputa "Aberto e Fechado".
  • Estratégias avançadas (Lance percentual, Sniper de último segundo).
  • Plataforma SaaS 100% em nuvem (V1 será instalável ou script via terminal).
  • Envio automático de documentos de habilitação.

Lógica e Fluxo de Decisão (Algoritmo)

Este é o fluxograma interno do Worker (robô) que executa a automação dos lances.

[INÍCIO]
  |
  +-- 1. Autenticar no Portal (Gov.br)
  |      - Verifica sucesso. Se erro -> Alerta Usuário e Para.
  |
  +-- 2. Selecionar Pregão Ativo e Item (Fornecido pelo DB)
  |
  +-- 3. Carregar Parâmetros:
  |      [VALOR_MINIMO_ACEITAVEL] = ex: R$ 100,00
  |      [DECREMENTO] = ex: R$ 0,05
  |
  +-- [LOOP DE MONITORAMENTO] <--+
         |                       |
         +-- Ler Tela/API: Qual o atual [MENOR_LANCE]?
         |
         +-- Condição A: O [MENOR_LANCE] é MEU?
         |      - SIM: Aguardar 2s. Retornar ao LOOP.
         |      - NÃO: Seguir em frente.
         |
         +-- Condição B: O status do item é "ABERTO" para lances?
         |      - NÃO: Sair do LOOP (Pregão pausado ou encerrado).
         |      - SIM: Seguir em frente.
         |
         +-- Calcular: [NOVO_LANCE] = [MENOR_LANCE] - [DECREMENTO]
         |
         +-- Condição C: [NOVO_LANCE] >= [VALOR_MINIMO_ACEITAVEL]?
         |      - NÃO: Bati na margem! Parar de dar lances. Notificar Usuário.
         |      - SIM: Preparar envio!
         |
         +-- AÇÃO: Enviar Requisição HTTP POST de Lance com [NOVO_LANCE]
         |
         +-- Verificar resposta do servidor (Sucesso/Recusado)
         |
         +-- Aguardar X segundos (evitar rate limit).
         |
         +-----------------------+ (Reinicia o Loop)

* Nota: Esta lógica é a base do MVP. Sistemas avançados incluem tratativas assíncronas via WebSockets para evitar polling excessivo.
